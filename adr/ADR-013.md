# ADR-013: Cognitive Runtime Architecture for AI Persona

## Date
2026-05-12

## Status
Accepted

## Context
AI conversation responses felt robotic and lacked human-like qualities. Simply optimizing prompts (prompt engineering approach) was insufficient — what was needed is a stateful cognitive system that maintains consistent persona, emotional state, relationship dynamics, and memory across conversations.

## Decision
Adopt a Cognitive Runtime Architecture built as a standalone engine under `src/lib/ai/persona/`:

### Core files
- `types.ts` — PersonaConfig, RuntimeState, RelationshipState, BehavioralPolicy, MemoryEvent
- `state.ts` — Runtime state engine: mood, energy, engagement, social openness, relationship drift
- `relationship.ts` — RelationshipState: familiarity, trust, comfort, humor_acceptance
- `memory.ts` — Event-sourced memory: episodic compression, structured extraction, retrieval by importance/category
- `policies.ts` — Behavior Modulation Layer: correction_style, initiative_level, humor_level, verbosity
- `mutation.ts` — Runtime State Mutation Rules engine (signal → state change)
- `orchestrator.ts` — Cognitive Pipeline Coordinator: memory retrieval → state merge → policy modulation → prompt assembly → response post-processing → memory extraction → state mutation
- `persona.ts` — Public facade for PersonaAgent
- `prompts.ts` — Dynamic prompt templates with persona/state injection
- `store.ts` — Event-sourced MongoDB: persona_profiles, relationship_states, memory_events, conversation_sessions

### Key architecture principles
1. **Stateful over stateless** — Persona maintains runtime state (mood, energy, engagement) that mutates based on conversation signals
2. **Relationship as first-class object** — relationship_state (familiarity/trust/comfort/humor_acceptance) is separate from relationship_history
3. **Event-sourced memory** — Raw logs for current session only; structured extractions (event, importance, category) persisted long-term
4. **Behavioral policies as modulation layer** — Policies sit between persona config + state to produce context-appropriate behavior
5. **Mutation rules drive state** — Explicit rules (e.g., short replies → engagement decay) ensure state changes are predictable and testable

### Integration strategy
P0: Build the core engine (all types, state, memory, policies, mutation, orchestration)
P1: Validate behavioral consistency (memory recall, tone, state drift, initiative quality)
P2: Integrate into conversation mode (API routes, AI adapters, store optimization)

## Consequences
+ AI responses will be driven by a consistent, stateful persona rather than raw prompts
+ Relationship state enables adaptive behavior (humor level, teaching intensity, initiative)
+ Event-sourced memory prevents token explosion while preserving important context
+ Explicit mutation rules make persona behavior testable and predictable
- Significant new code (~10 files) in a dedicated persona subdirectory
- Runtime state and mutation rules add complexity to the AI pipeline
- Existing AiAdapter interface needs updating to support persona context
