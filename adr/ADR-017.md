# ADR-017: Mutual Modulation Architecture

## Status

Accepted (2026-05-13)

## Context

Phase 11 (Conversation Intelligence) needed to integrate two previously independent subsystems:
1. **Conversation Pipeline** (`conversation/pipeline.ts`) ā€” intent classification, response budget, momentum tracking
2. **Persona Engine** (`persona/`) ā€” emotional state, relationship tracking, behavioral policies

Each subsystem worked independently. The conversation pipeline didn't influence the persona engine, and the persona state didn't influence the conversation budget.

## Decision

Integrate both subsystems through `PersonaAgent.processTurn()` with a 6-stage mutual modulation loop:

1. Build `PersonaModulationInput` from current `runtime.engagement`, `policy.verbosity`, `relationship.familiarity`
2. Run `runPipeline()` ā†’ produces `intent`, `budget`, `momentum`
3. Bridge via `deriveSignalFromIntent()` ā€” 7 intentā†’signal mappings
4. Feed derived signal into `runCognitivePipeline()` ā†’ updates runtime/relationship
5. Updated runtime (valence/arousal/engagement) feeds back into next turn's modulation
6. Momentum state persisted in `PersonaAgent` across turns

## Consequences

- **Mutual feedback loop**: intent affects emotion (via signal), emotion affects budget (via modulation)
- **Backward compatible**: signal-only `processTurn()` (without `userMessage`) skips conversation pipeline
- **Explicit integration surface**: `deriveSignalFromIntent()` in `state.ts` is the single bridge function
- **Testable**: 19 integration tests covering signal derivation, momentum tracking, intentOverride, and emotional impact
