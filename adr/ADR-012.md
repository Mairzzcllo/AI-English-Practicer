# ADR-012: Screen Component Pattern for Phase-Based Pages

## Date
2026-05-10

## Status
Accepted

## Context
`interview/page.tsx` had three distinct phases (`intro`, `conversing`, `ended`) rendered via inline JSX. The file grew to 451 lines, mixing orchestration logic (hooks, state, callbacks) with UI markup.

## Decision
Extract each phase into a dedicated screen component under `src/components/ui/`:
- `IntroScreen` — language select, speed select, start button, error/mic permission display
- `ConversingScreen` — sidebar (badge, timer, end button), message list, real-time transcript, status bar
- `EndedScreen` — score circle, summary, stats grid, navigation buttons

Each screen receives all data via props — no embedded state or hook calls. The page is reduced to a lightweight orchestration layer that wires hooks to props and switches screens based on `phase` state.

## Consequences
+ Page.tsx at 268 lines (was 451), purely orchestration
+ UI markup lives in focused, testable screen components (IntroScreen: 17 tests, ConversingScreen: 20 tests, EndedScreen: 10 tests)
+ New phases can be added without touching other screens
- Prop drilling increases for ConversingScreen (14 props) — acceptable since it's a terminal consumer
